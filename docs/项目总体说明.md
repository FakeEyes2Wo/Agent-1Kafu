# 项目总体说明

## 1. 项目目标

本项目用于构建一个面向电商客服场景的多模态智能体系统。系统基于举办方提供的手册文本、手册插图和公开问题数据，完成知识库检索、客服回答生成、API 服务封装和比赛提交文件生成。

项目遵循以下原则：

- 使用 `uv` 管理 Python 环境和依赖。
- 使用 `.env` 统一管理模型、接口密钥、路径和检索参数。
- 使用 LangChain 调用 OpenAI 兼容模型与 embedding。
- 使用 LangGraph 编排客服 Agent 流程。
- 不做复杂命令行工具，只提供普通 Python 脚本。
- 代码保持最小化，优先保证比赛可运行、可复现、可提交。

## 2. 核心能力

系统支持以下能力：

- 读取 `data/手册/*.txt` 构建本地 RAG 知识库。
- 解析手册中的 `<PIC>` 标记和图片 ID，回答中保留相关图片提示。
- 支持文本问题和可选 Base64 图片输入。
- 通过 `/chat` 提供 RESTful API 服务。
- 支持会话 ID，多轮对话可读取最近历史。
- 针对公开题 `data/question_public.csv` 生成根目录 `submission.csv`。
- 参考 `data/submission_example.csv` 的提交格式，输出两列：`id,ret`。

其中：

- `src/kefu_agent/api.py`：FastAPI `/chat` 服务。

- `scripts/run_api.py`：启动 API 服务。
- `scripts/generate_submission.py`：生成比赛提交文件。

## 4. 配置说明

项目使用 `.env` 管理配置。首次使用时复制样例文件：

```powershell
Copy-Item .env.example .env
```

主要配置项：

```env
KAFU_API_TOKEN=change-me

OPENAI_API_KEY=your-api-key
OPENAI_BASE_URL=https://api.openai.com/v1
CHAT_MODEL=gpt-4o-mini
VISION_MODEL=gpt-4o-mini
VISION_MODEL_URL=https://api.openai.com/v1
EMBEDDING_MODEL=Qwen/Qwen3-Embedding-0.6B
EMBEDDING_BACKEND=sentence_transformers
EMBEDDING_QUERY_PROMPT_NAME=query
EMBEDDING_MODEL_DIR=./storage/models

DATA_DIR=./data
MANUAL_DIR=./data/手册
IMAGE_DIR=./data/手册/插图
VECTORSTORE_DIR=./storage/vectorstore
SESSION_STORE_DIR=./storage/sessions

CHUNK_SIZE=700
CHUNK_OVERLAP=120
TOP_K=8
MAX_HISTORY_TURNS=6
```

答案生成必须配置有效的 `OPENAI_API_KEY` 并由 `CHAT_MODEL` 完成，不再使用模板回答兜底。检索默认使用 Hugging Face `Qwen/Qwen3-Embedding-0.6B`，通过 `sentence-transformers` 本地生成向量；如需调用 OpenAI 兼容 embedding 接口，可设置 `EMBEDDING_BACKEND=openai`。

## 5. 运行方式

安装依赖：

```powershell
uv sync
```

构建索引：

```powershell
uv run python scripts/build_index.py
```

启动 API：

```powershell
uv run python scripts/run_api.py
```

生成比赛提交：

```powershell
uv run python scripts/generate_submission.py
```

生成后根目录会出现：

```text
submission.csv
```

该文件格式与 `data/submission_example.csv` 一致：

```csv
id,ret
1,您好，...
```

## 6. 设计取舍

为了保持实现最小化，第一版没有引入复杂数据库、外部向量服务、Web 前端、在线微调或复杂 CLI。索引以 JSONL 形式持久化到 `storage/vectorstore/index.jsonl`，会话历史以 JSON 文件保存在 `storage/sessions/`。

这种设计便于比赛环境快速复现，也方便后续按需要替换为 Chroma、FAISS、Milvus 或更复杂的 rerank 流程。
